# HomeTube — Follow-Up Plan: Closing Plan Gaps

This is the follow-up to [`1778451852595-cosmic-panda.md`](1778451852595-cosmic-panda.md) (the
main implementation plan). The main plan has been implemented across 20
phases (commits `3f5705e`..`d1bbf72` on `main`), but a thorough audit
identified items that are partially implemented or deliberately reduced.
This document captures those gaps so a future agent session can finish
them.

- Branch `main` at `d1bbf72`, pushed to `git@github.com:jimsimon/hometube.git`.
- All 28 database tables, every API endpoint in the original plan, and
  every Phase 0–19 functional item are implemented and committed.
- Local checks all pass:
  - `cargo fmt --check` clean
  - `cargo clippy --all-targets --all-features -- -D warnings` clean
  - 215 backend tests pass
  - `npx oxlint`, `npx oxfmt --check`, `npx tsc --noEmit` clean

  - 3 Playwright tests pass against the live release binary
- CI workflow exists at `.github/workflows/ci.yml` (lint → test+coverage
  → build → e2e+a11y → publish-docker).

### T-11. Migrate offline-video storage from Cache API to OPFS

**Priority:** Medium (plan fidelity + correctness for big files)
**Plan reference:** Architecture diagram, Project Structure
(`frontend/src/services/offline.ts # OPFS + service worker helpers`),
Phase 1 service worker section, "Offline Video Download Flow", and
the Limitations note that explicitly cites OPFS.
**Status:** DIVERGENT — current implementation stores downloaded
videos in `caches.open(VIDEO_CACHE)` instead of OPFS. Decision was
made by the F1-F3 sub-agent under guidance that "Cache API was
acceptable", but the plan was explicit about OPFS.

**Why the plan called for OPFS, and why it matters:**

1. **Streamed writes.** OPFS's `FileSystemWritableFileStream`
   accepts a `ReadableStream` directly via `pipeTo()`, so a 2 GB
   1080p download writes to disk as it arrives. The Cache API
   requires the full body be buffered in memory before
   `cache.put(request, response)` returns, which is fine for ~500 MB
   files on a desktop but can OOM a low-RAM tablet.
2. **Per-file size accounting.** OPFS gives `FileSystemFileHandle`
   →  `getFile()` → `.size` for each download. Cache API only
   gives aggregate `navigator.storage.estimate()`; per-video sizes
   need extra bookkeeping in `localStorage`.
3. **Random-access seeks.** `FileSystemSyncAccessHandle.read(buf,
   { at })` is direct. With Cache API, range requests against a
   single cached `Response` work in modern Chromium but the SW
   has to slice the body manually for Firefox/Safari.
4. **Background Fetch sink.** When we eventually wire Background
   Fetch (Chromium only — currently in "Items deliberately not in
   this plan"), OPFS is the natural place for it to deposit bytes;
   Cache API is awkward for that flow.

**The trade-off we accept:** OPFS-from-service-worker requires a
bridge. Service workers cannot access OPFS directly in current
specs — only window/worker contexts can. The standard pattern is
to keep a long-lived `SharedWorker` (or a dedicated client) that
the SW posts requests to via `postMessage`/`BroadcastChannel`.
That bridge is non-trivial and Safari's SharedWorker support is
shaky. We mitigate by (a) using a dedicated client (one of the
open tabs/PWA windows) as the OPFS proxy when present, and (b)
falling back to "online-only" for downloaded videos when no
proxy client is available — the plan accepts this kind of
graceful degradation.

**Files to create:**
- `frontend/src/services/offline-opfs.ts` — replaces the Cache-API
  implementation in `offline.ts`. Same public surface
  (`saveVideoToOpfs`, `getOfflineVideoStream`, `listOfflineVideos`,
  `deleteOfflineVideo`, `estimateStorage`) but backed by:
  - `navigator.storage.getDirectory()` →
    `getDirectoryHandle('hometube-videos', { create: true })` →
    per-video subdirectory keyed by `<videoId>/<quality>.mp4`.
  - Streamed writes via
    `(await fileHandle.createWritable()).pipeTo(response.body)`.
  - A small `index.json` (or `localStorage` shadow) keeping
    `{ videoId, quality, size, downloadedAt, title, thumbnailUrl,
    channelTitle, duration }` for `listOfflineVideos`.
- `frontend/src/services/opfs-bridge.ts` — message protocol the
  service worker uses to ask any open client for an OPFS read.
  Supported messages:
  - `{ type: 'opfs.read', videoId, quality, range?: [start, end] }`
    → client returns `{ type: 'opfs.read.response', body: Uint8Array,
    contentType, contentRange? }`.
  - `{ type: 'opfs.exists', videoId, quality }` → boolean.
  - The bridge is registered in `sw-register.ts` so every page
    loads it automatically.

**Files to update:**
- `frontend/src/sw.ts` — replace the Cache-API video-proxy
  intercept with the OPFS-bridge flow:
  1. On `fetch` for `/api/proxy/video?...` or `/api/downloads/:id/stream?...`,
     extract `videoId` + `quality` from the URL.
  2. Ask all `clients.matchAll({ includeUncontrolled: true,
     type: 'window' })` whether any has the file via `opfs.exists`.
  3. If yes, request `opfs.read` (with the inbound `Range` header
     forwarded), wrap the returned bytes in a `Response` with the
     correct `Content-Type`, `Content-Length`, `Content-Range` (for
     206 responses) and `Accept-Ranges: bytes`.
  4. If no client can serve it (no tabs open, or none has the
     file): fall through to the network. If offline AND no client
     can serve, return the offline-fallback page.
- `frontend/src/services/offline.ts` — keep the file as a thin
  re-export shim that delegates to `offline-opfs.ts` so existing
  callers (`offline-downloads-list.ts`, `video-player.ts` download
  button) keep working without churn. Update the doc comment to
  document the shift.
- `frontend/src/components/video-player.ts` — when triggering a
  download, switch from the buffer-then-`cache.put` pattern to
  direct streamed write into OPFS. The download button's progress
  indicator can now use `loaded`/`total` from the underlying
  `pipeTo()` strategy.
- `frontend/src/components/offline-downloads-list.ts` — if the
  list pulls per-video sizes, switch from the Cache-API count to
  OPFS file sizes.

**Migration considerations:**
- Any existing user with downloads in the Cache API needs
  migration. Easiest: on first run of the new code, scan the old
  cache and copy each entry into OPFS, then delete the old cache.
  Implement this as a one-shot in `sw-register.ts` guarded by a
  `localStorage.getItem('hometube-opfs-migrated')` flag.
- Safari Private Browsing disables OPFS entirely. Detect this
  (`'storage' in navigator && navigator.storage.getDirectory` is
  the right capability check) and fall back to "downloads
  unavailable" in that mode — the original plan already calls this
  out as a Limitation.

## Items deliberately not in this plan

These are documented elsewhere (commit messages, audit report) as
acceptable substitutions or design decisions. They do **not** need
to be implemented to consider the plan complete.

- **PIN hashed with argon2 instead of bcrypt** — the original plan was
  internally inconsistent here: the prose in a few places mentioned
  bcrypt, but the canonical dependency list explicitly specified
  `argon2 = "0.5"` with the comment "PIN hashing (recommended over
  bcrypt)" and the version-constraint note "Even bcrypt's maintainers
  recommend argon2 for new projects." The implementation followed the
  dependency list (see `src/routes/auth.rs::hash_pin` /
  `verify_pin`, which use `Argon2::default()` — Argon2id with the
  crate's OWASP-aligned defaults). No further work needed; this entry
  exists only to flag the prose-vs-deps inconsistency for future
  readers of the original plan.
- **Background Fetch API** for downloads — Chromium-only enhancement;
  current implementation uses standard `fetch` while the page is open.
  Documented in `services/offline.ts`. With OPFS now the underlying
  sink (T-11), Background Fetch can be wired in later by pointing the
  fetch sink at a `FileSystemWritableFileStream` and reusing the
  existing manifest-update path.
