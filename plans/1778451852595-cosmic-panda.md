# HomeTube - Implementation Plan

## Overview

**HomeTube** — a self-hosted YouTube frontend for kids that allows parents to control what content their children can access, while syncing children's activity (likes, subscriptions) back to their real YouTube accounts.

- **Repository:** `github.com/<your-username>/hometube`
- **License:** AGPL-3.0 (open source, self-hostable, protects against closed-source SaaS forks)
- **Visibility:** Public

---

## UI/UX Principles

1. **Mobile-first, responsive design** — All pages must work well on phone, tablet, and desktop. Use CSS Grid/Flexbox with responsive breakpoints. No separate "mobile" or "desktop" builds.

2. **Lightweight: plain HTML first, Web Awesome when needed** — Prefer semantic HTML elements (`<nav>`, `<main>`, `<section>`, `<button>`, `<details>`, etc.) over framework components wherever possible. Only reach for Web Awesome components (`<wa-dialog>`, `<wa-dropdown>`, `<wa-tab-group>`, etc.) when they provide functionality that plain HTML cannot. Minimize JS and CSS payload.

3. **Accessibility (a11y) as a first-class concern:**
   - All interactive elements must be keyboard navigable
   - Proper ARIA attributes where semantic HTML alone isn't sufficient
   - Focus management on navigation and modal transitions
   - Color contrast ratios meet WCAG 2.1 AA minimum (4.5:1 for text)
   - Screen reader testing for all user flows
   - Automated a11y testing in CI (see below)

4. **All pages use `<wa-page>` as root layout** — Provides consistent theme support (light/dark/system) and page structure.

5. **Progressive enhancement** — Core content is visible without JavaScript where possible. Interactive features enhance the base experience.

## Architecture

```
┌─────────────────────────────────────────────────────┐
│                    Client (Browser)                   │
│  Lit Web Components + TypeScript + Web Awesome       │
│  Video Player: vidstack                              │
│  Service Worker: OPFS offline + segment caching      │
├─────────────────────────────────────────────────────┤
│                 Rust Backend (Axum)                   │
│  ┌──────────┐ ┌──────────┐ ┌─────────────────────┐ │
│  │ HTML     │ │ API      │ │ Video Proxy         │ │
│  │ Pages    │ │ Routes   │ │ (reqwest streaming) │ │
│  │ (askama) │ │ (JSON)   │ │                     │ │
│  └──────────┘ └──────────┘ └─────────────────────┘ │
│  ┌──────────┐ ┌──────────┐ ┌─────────────────────┐ │
│  │ OAuth2   │ │ YouTube  │ │ Static Assets       │ │
│  │ Flow     │ │ Data API │ │ (JS/CSS bundles)    │ │
│  └──────────┘ └──────────┘ └─────────────────────┘ │
│  ┌──────────┐ ┌──────────┐                         │
│  │ yt-dlp   │ │ Cron     │                         │
│  │ Subprocess│ │ Scheduler│                         │
│  └──────────┘ └──────────┘                         │
├─────────────────────────────────────────────────────┤
│              SQLite Database (sqlx)                   │
└─────────────────────────────────────────────────────┘
```

**Routing model: Multi-Page App (MPA)**
- Axum serves HTML pages rendered by askama templates
- Each page includes only the Lit web components it needs via `<script type="module">`
- Navigation uses standard `<a>` links (full page loads)
- Vite bundles Lit components into ES modules (served as static assets)
- Lightest possible JS payload per page

**Version constraint notes:**
- `hmac 0.12`, `sha2 0.10`, `rand 0.8`: Pinned to these major versions because `openidconnect` depends on them. Using newer majors (0.13/0.11/0.10) would cause version conflicts.
- `argon2` chosen over `bcrypt`: Even bcrypt's maintainers recommend argon2 for new projects.

- `thiserror` + `anyhow` used together: `thiserror` for the app's error enum (implements `IntoResponse`), `anyhow` for internal error propagation.

## Database Schema

```sql
-- Accounts table (both parents and children)
CREATE TABLE accounts (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    google_id TEXT UNIQUE NOT NULL,
    email TEXT NOT NULL,
    display_name TEXT NOT NULL,
    avatar_url TEXT,
    account_type TEXT NOT NULL CHECK (account_type IN ('parent', 'child')),
    pin_hash TEXT,                    -- bcrypt hash of PIN (required for parents, optional for children)
    access_token TEXT NOT NULL,
    refresh_token TEXT NOT NULL,
    token_expires_at INTEGER NOT NULL,
    created_at INTEGER NOT NULL DEFAULT (unixepoch()),
    updated_at INTEGER NOT NULL DEFAULT (unixepoch())
);

-- Sessions
CREATE TABLE sessions (
    id TEXT PRIMARY KEY,  -- random session ID
    account_id INTEGER NOT NULL REFERENCES accounts(id),
    expires_at INTEGER NOT NULL,
    created_at INTEGER NOT NULL DEFAULT (unixepoch())
);

-- Allowlisted channels per child
CREATE TABLE allowlisted_channels (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    child_account_id INTEGER NOT NULL REFERENCES accounts(id),
    channel_id TEXT NOT NULL,        -- YouTube channel ID
    channel_title TEXT NOT NULL,
    channel_thumbnail_url TEXT,
    added_by INTEGER NOT NULL REFERENCES accounts(id),  -- parent who added it
    created_at INTEGER NOT NULL DEFAULT (unixepoch()),
    UNIQUE(child_account_id, channel_id)
);

-- Allowlisted playlists per child
CREATE TABLE allowlisted_playlists (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    child_account_id INTEGER NOT NULL REFERENCES accounts(id),
    playlist_id TEXT NOT NULL,       -- YouTube playlist ID
    playlist_title TEXT NOT NULL,
    playlist_thumbnail_url TEXT,
    added_by INTEGER NOT NULL REFERENCES accounts(id),
    created_at INTEGER NOT NULL DEFAULT (unixepoch()),
    UNIQUE(child_account_id, playlist_id)
);

-- Allowlisted individual videos per child
CREATE TABLE allowlisted_videos (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    child_account_id INTEGER NOT NULL REFERENCES accounts(id),
    video_id TEXT NOT NULL,          -- YouTube video ID
    video_title TEXT NOT NULL,
    video_thumbnail_url TEXT,
    channel_title TEXT,
    added_by INTEGER NOT NULL REFERENCES accounts(id),
    created_at INTEGER NOT NULL DEFAULT (unixepoch()),
    UNIQUE(child_account_id, video_id)
);

-- Blocked videos per child (overrides all allowlists)
CREATE TABLE blocked_videos (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    child_account_id INTEGER NOT NULL REFERENCES accounts(id),
    video_id TEXT NOT NULL,
    video_title TEXT,
    blocked_by INTEGER NOT NULL REFERENCES accounts(id),
    reason TEXT,
    created_at INTEGER NOT NULL DEFAULT (unixepoch()),
    UNIQUE(child_account_id, video_id)
);

-- Per-child settings (parent-controlled)
CREATE TABLE child_settings (
    child_account_id INTEGER PRIMARY KEY REFERENCES accounts(id),
    downloads_enabled INTEGER NOT NULL DEFAULT 1,   -- 1 = child can download, 0 = disabled
    max_quality TEXT DEFAULT NULL,                   -- NULL = no cap, or '480p', '720p', '1080p'
    playback_speed_locked INTEGER NOT NULL DEFAULT 0, -- 1 = child cannot change playback speed
    autoplay_enabled INTEGER NOT NULL DEFAULT 1,    -- 1 = videos auto-play next, 0 = disabled
    autoplay_max_consecutive INTEGER DEFAULT NULL,  -- NULL = unlimited, or max videos in a row before pause
    created_at INTEGER NOT NULL DEFAULT (unixepoch()),
    updated_at INTEGER NOT NULL DEFAULT (unixepoch())
);

-- Usage limits per child per day of week
CREATE TABLE usage_limits (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    child_account_id INTEGER NOT NULL REFERENCES accounts(id),
    day_of_week INTEGER NOT NULL CHECK (day_of_week BETWEEN 0 AND 6), -- 0=Sunday
    max_hours REAL NOT NULL DEFAULT 2.0,
    allowed_start_time TEXT NOT NULL DEFAULT '08:00', -- HH:MM format
    allowed_end_time TEXT NOT NULL DEFAULT '20:00',
    created_at INTEGER NOT NULL DEFAULT (unixepoch()),
    updated_at INTEGER NOT NULL DEFAULT (unixepoch()),
    UNIQUE(child_account_id, day_of_week)
);

-- Usage tracking (actual time spent watching)
CREATE TABLE usage_log (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    child_account_id INTEGER NOT NULL REFERENCES accounts(id),
    video_id TEXT NOT NULL,
    started_at INTEGER NOT NULL,
    ended_at INTEGER,
    duration_seconds INTEGER
);

-- Child's playlists (2-way synced with YouTube)
CREATE TABLE child_playlists (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    child_account_id INTEGER NOT NULL REFERENCES accounts(id),
    youtube_playlist_id TEXT,           -- YouTube playlist ID (NULL until first sync for app-created)
    title TEXT NOT NULL,
    description TEXT,
    is_own INTEGER NOT NULL DEFAULT 1,  -- 1 = created by child, 0 = added to library
    source TEXT NOT NULL DEFAULT 'app' CHECK (source IN ('app', 'youtube')),
    sync_status TEXT NOT NULL DEFAULT 'synced' CHECK (sync_status IN ('synced', 'pending_create', 'pending_update', 'pending_delete', 'error')),
    is_deleted INTEGER NOT NULL DEFAULT 0,
    created_at INTEGER NOT NULL DEFAULT (unixepoch()),
    updated_at INTEGER NOT NULL DEFAULT (unixepoch())
);

-- Videos in child playlists
CREATE TABLE child_playlist_videos (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    playlist_id INTEGER NOT NULL REFERENCES child_playlists(id) ON DELETE CASCADE,
    video_id TEXT NOT NULL,
    video_title TEXT NOT NULL,
    video_thumbnail_url TEXT,
    channel_title TEXT,
    position INTEGER NOT NULL,
    added_at INTEGER NOT NULL DEFAULT (unixepoch()),
    UNIQUE(playlist_id, video_id)
);

-- Child subscriptions (2-way synced with YouTube)
CREATE TABLE child_subscriptions (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    child_account_id INTEGER NOT NULL REFERENCES accounts(id),
    channel_id TEXT NOT NULL,
    channel_title TEXT NOT NULL,
    channel_thumbnail_url TEXT,
    youtube_subscription_id TEXT,    -- YouTube's subscription resource ID (for delete operations)
    source TEXT NOT NULL DEFAULT 'app' CHECK (source IN ('app', 'youtube')),  -- where it originated
    sync_status TEXT NOT NULL DEFAULT 'synced' CHECK (sync_status IN ('synced', 'pending_push', 'pending_delete', 'error')),
    is_deleted INTEGER NOT NULL DEFAULT 0,  -- soft delete (for tracking removals)
    subscribed_at INTEGER NOT NULL DEFAULT (unixepoch()),
    updated_at INTEGER NOT NULL DEFAULT (unixepoch()),
    UNIQUE(child_account_id, channel_id)
);

-- Watch history (for "continue watching")
CREATE TABLE watch_history (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    child_account_id INTEGER NOT NULL REFERENCES accounts(id),
    video_id TEXT NOT NULL,
    video_title TEXT NOT NULL,
    video_thumbnail_url TEXT,
    channel_title TEXT,
    duration_seconds INTEGER,        -- total video duration
    progress_seconds INTEGER NOT NULL DEFAULT 0,  -- where they left off
    last_watched_at INTEGER NOT NULL DEFAULT (unixepoch()),
    UNIQUE(child_account_id, video_id)
);

-- Video likes (2-way synced with YouTube)
CREATE TABLE video_likes (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    child_account_id INTEGER NOT NULL REFERENCES accounts(id),
    video_id TEXT NOT NULL,
    video_title TEXT,
    video_thumbnail_url TEXT,
    source TEXT NOT NULL DEFAULT 'app' CHECK (source IN ('app', 'youtube')),
    sync_status TEXT NOT NULL DEFAULT 'synced' CHECK (sync_status IN ('synced', 'pending_push', 'pending_delete', 'error')),
    is_deleted INTEGER NOT NULL DEFAULT 0,
    liked_at INTEGER NOT NULL DEFAULT (unixepoch()),
    updated_at INTEGER NOT NULL DEFAULT (unixepoch()),
    UNIQUE(child_account_id, video_id)
);

-- Sync state tracking (last sync cursor per account per data type)
CREATE TABLE sync_state (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    account_id INTEGER NOT NULL REFERENCES accounts(id),
    data_type TEXT NOT NULL CHECK (data_type IN ('subscriptions', 'likes', 'playlists')),
    last_synced_at INTEGER,
    last_page_token TEXT,           -- YouTube API pagination token (for incremental sync)
    etag TEXT,                      -- YouTube API ETag (for change detection)
    UNIQUE(account_id, data_type)
);

-- Setup state
CREATE TABLE app_config (
    key TEXT PRIMARY KEY,
    value TEXT NOT NULL
);

-- Cron jobs
CREATE TABLE cron_jobs (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT UNIQUE NOT NULL,
    description TEXT,
    job_type TEXT NOT NULL,          -- e.g., 'ytdlp_update', 'youtube_sync' (maps to handler function)
    schedule TEXT NOT NULL,          -- cron expression (e.g., "0 3 * * *")
    schedule_preset TEXT,            -- human-readable label (e.g., "Every hour", "Daily (3:00 AM)")
    allowed_presets TEXT NOT NULL,   -- JSON array of allowed preset labels for this job type
    enabled INTEGER NOT NULL DEFAULT 1,
    last_run_at INTEGER,
    last_run_status TEXT CHECK (last_run_status IN ('success', 'failure', NULL)),
    last_run_message TEXT,
    next_run_at INTEGER,
    created_at INTEGER NOT NULL DEFAULT (unixepoch())
);

-- Cron job run history
CREATE TABLE cron_job_runs (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    job_id INTEGER NOT NULL REFERENCES cron_jobs(id),
    started_at INTEGER NOT NULL,
    finished_at INTEGER,
    status TEXT NOT NULL CHECK (status IN ('running', 'success', 'failure')),
    message TEXT,
    output TEXT                      -- stdout/stderr from the job
);

-- yt-dlp version tracking
CREATE TABLE ytdlp_info (
    id INTEGER PRIMARY KEY CHECK (id = 1),  -- singleton row
    current_version TEXT,
    last_checked_at INTEGER,
    last_updated_at INTEGER,
    binary_path TEXT NOT NULL
);

-- Server-side segment cache index
CREATE TABLE segment_cache (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    video_id TEXT NOT NULL,
    format_id TEXT NOT NULL,           -- yt-dlp format identifier (e.g., "137", "251")
    segment_number INTEGER NOT NULL,
    file_path TEXT NOT NULL,           -- path on server disk
    file_size_bytes INTEGER NOT NULL,
    cached_at INTEGER NOT NULL DEFAULT (unixepoch()),
    last_accessed_at INTEGER NOT NULL DEFAULT (unixepoch()),
    UNIQUE(video_id, format_id, segment_number)
);

-- Index for LRU eviction queries
CREATE INDEX idx_segment_cache_lru ON segment_cache(last_accessed_at ASC);

-- Video metadata cache (tracks which videos have been extracted)
CREATE TABLE video_metadata_cache (
    video_id TEXT PRIMARY KEY,
    metadata_json TEXT NOT NULL,       -- full yt-dlp JSON output (formats, thumbnails, etc.)
    dash_manifest TEXT,                -- rewritten DASH manifest XML
    cached_at INTEGER NOT NULL DEFAULT (unixepoch()),
    expires_at INTEGER NOT NULL        -- cached_at + TTL (default 4 hours)
);

-- Search term log (for parent visibility into what kids search for)
CREATE TABLE search_log (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    child_account_id INTEGER NOT NULL REFERENCES accounts(id),
    query TEXT NOT NULL,
    result_count INTEGER NOT NULL DEFAULT 0,
    searched_at INTEGER NOT NULL DEFAULT (unixepoch())
);

-- Video bookmarks (child can bookmark timestamps in videos)
CREATE TABLE video_bookmarks (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    child_account_id INTEGER NOT NULL REFERENCES accounts(id),
    video_id TEXT NOT NULL,
    video_title TEXT,
    timestamp_seconds INTEGER NOT NULL,
    label TEXT,                        -- optional user-provided label
    created_at INTEGER NOT NULL DEFAULT (unixepoch()),
    UNIQUE(child_account_id, video_id, timestamp_seconds)
);

-- Shared family playlists (parent-created, visible to selected children)
CREATE TABLE family_playlists (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    created_by INTEGER NOT NULL REFERENCES accounts(id),  -- parent who created it
    title TEXT NOT NULL,
    description TEXT,
    created_at INTEGER NOT NULL DEFAULT (unixepoch()),
    updated_at INTEGER NOT NULL DEFAULT (unixepoch())
);

-- Which children can see a family playlist
CREATE TABLE family_playlist_members (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    playlist_id INTEGER NOT NULL REFERENCES family_playlists(id) ON DELETE CASCADE,
    child_account_id INTEGER NOT NULL REFERENCES accounts(id),
    UNIQUE(playlist_id, child_account_id)
);

-- Videos in family playlists
CREATE TABLE family_playlist_videos (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    playlist_id INTEGER NOT NULL REFERENCES family_playlists(id) ON DELETE CASCADE,
    video_id TEXT NOT NULL,
    video_title TEXT NOT NULL,
    video_thumbnail_url TEXT,
    channel_title TEXT,
    position INTEGER NOT NULL,
    added_at INTEGER NOT NULL DEFAULT (unixepoch()),
    UNIQUE(playlist_id, video_id)
);

-- Parent notifications (in-app notification queue)
CREATE TABLE parent_notifications (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    parent_account_id INTEGER NOT NULL REFERENCES accounts(id),
    notification_type TEXT NOT NULL CHECK (notification_type IN (
        'time_limit_approaching', 'time_limit_reached',
        'ytdlp_failure', 'sync_error', 'token_expired',
        'new_search_term', 'system_update'
    )),
    title TEXT NOT NULL,
    message TEXT NOT NULL,
    metadata TEXT,                     -- JSON blob with extra context (child_id, video_id, etc.)
    is_read INTEGER NOT NULL DEFAULT 0,
    created_at INTEGER NOT NULL DEFAULT (unixepoch())
);

-- Sleep timers (active timers set by children)
CREATE TABLE sleep_timers (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    child_account_id INTEGER NOT NULL REFERENCES accounts(id),
    timer_type TEXT NOT NULL CHECK (timer_type IN ('after_video', 'minutes')),
    minutes_remaining INTEGER,         -- for 'minutes' type
    videos_remaining INTEGER DEFAULT 1, -- for 'after_video' type (always 1 = after this video)
    started_at INTEGER NOT NULL DEFAULT (unixepoch()),
    expires_at INTEGER,                -- calculated end time
    is_active INTEGER NOT NULL DEFAULT 1,
    UNIQUE(child_account_id, is_active)  -- only one active timer per child
);

-- Offline download requests (tracks what children have downloaded client-side)
CREATE TABLE offline_downloads (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    child_account_id INTEGER NOT NULL REFERENCES accounts(id),
    video_id TEXT NOT NULL,
    video_title TEXT NOT NULL,
    video_thumbnail_url TEXT,
    channel_title TEXT,
    quality_label TEXT NOT NULL,
    file_size_bytes INTEGER,
    duration_seconds INTEGER,
    status TEXT NOT NULL DEFAULT 'pending' CHECK (status IN ('pending', 'downloading', 'complete', 'failed', 'deleted')),
    downloaded_at INTEGER,
    UNIQUE(child_account_id, video_id, quality_label)
);

    If child account: no PIN required, switch immediately →
  Session cookie set for selected account →
  Redirect to appropriate home page (parent or child)
```

**PIN rules:**
- Parents MUST set a PIN during setup (prevents children from accessing admin)
- PINs are 4-6 digit numeric codes, stored as bcrypt hashes
- Children do NOT require PINs (frictionless access for kids)
- Failed PIN attempts: no lockout (kids might accidentally try), but log attempts
- Profile switching doesn't require re-authenticating with Google (session stays valid)
- Logging out returns to profile picker (not Google sign-in)

### Theme System (Light / Dark / System)
- All pages wrapped in Web Awesome's `<wa-page>` component (provides built-in theme support)
- Three modes available: **Light**, **Dark**, **System** (follows OS `prefers-color-scheme`)
- Theme toggle control in the nav bar (accessible from any page, for both parent and child)
- Preference stored per-profile in `localStorage` (persists across sessions per device)
- Default: "System" (respects the device's OS-level light/dark setting)
- `<wa-page>` handles CSS custom property switching automatically — no manual theming needed
- vidstack player inherits the theme (dark/light player chrome matches app theme)

**No-ads, no-comments policy:**
- The app NEVER fetches or displays YouTube comments (safety feature)
- Since video is served via yt-dlp proxy, no YouTube ads are served
- No recommended videos from YouTube's algorithm — only allowlisted content appears
- These are inherent architectural benefits, not toggles

### Captions & Subtitles
- yt-dlp extracts available subtitle tracks with `--list-subs` / included in JSON output
- Server converts subtitle formats to WebVTT (vidstack's native format)
- Captions are served via `/api/videos/:videoId/captions/:lang`
- vidstack player displays caption menu when tracks are available
- Useful for: accessibility, reading practice, quiet viewing times

### Audio-Only Mode
- DASH streams separate video and audio tracks natively
- Audio-only mode: player only requests audio segments (no video segments fetched)
- Saves bandwidth significantly (~128kbps audio vs ~5Mbps video+audio)
- UI: toggle in player controls to switch to audio-only (shows album art / thumbnail)
- Useful for: music content, podcasts, background listening, saving bandwidth
- Audio segments are cached independently in the segment cache

### Parental Preview
- Before adding content to a child's allowlist, parents can preview it
- Preview endpoints bypass the allowlist check (parent-only, uses parent's own session)
- Shows full video player so parent can watch and verify content is appropriate
- Parent can preview → then immediately add to allowlist from the same UI
- Channel/playlist preview shows recent videos with thumbnails for quick scanning

### Content Access Control Flow
```
User requests video → 
  Is user logged in? → No → Redirect to login
  Is video blocked for this child? → Yes → Deny (403)
  Is video in allowlisted videos? → Yes → Allow
  Is video from an allowlisted channel? → Yes → Allow
  Is video in an allowlisted playlist? → Yes → Allow
  → Deny (403)
```

```
Client requests /api/videos/:id/stream →
  Server calls yt-dlp to get formats →

```

### Two-Way YouTube Sync

**Outbound (App → YouTube):**
When a child performs an action in our app:
1. Action is recorded locally in SQLite (immediate UI response)
2. Background task syncs to YouTube API using child's OAuth tokens
3. If sync fails (rate limit, token expired), retry with exponential backoff
4. Update `sync_status` on the local record

**Inbound (YouTube → App):**
A periodic cron job (hourly) polls each child's YouTube account:
1. `Subscriptions.list(mine=true)` → diff against local `child_subscriptions`
2. `PlaylistItems.list(playlistId=LL)` → diff against local `video_likes`
3. `Playlists.list(mine=true)` → diff against local `child_playlists`
4. For each playlist: `PlaylistItems.list` → diff against local `child_playlist_videos`

**Diff logic:**
- New items on YouTube not in local DB → insert locally
- Items in local DB not on YouTube → mark as deleted locally (soft delete)
- Items that exist in both → update metadata if changed (title, thumbnail, etc.)

**Allowlist enforcement on inbound sync:**
- Data from YouTube that doesn't match the allowlist is stored in DB but **hidden** from child UI
- A `visible` or `allowlisted` flag is computed at query time (JOIN against allowlist tables)
- If a parent later allowlists the channel/video/playlist, the data appears automatically
- This preserves the child's full YouTube state for when they "graduate" to real YouTube

**Conflict resolution (concurrent edits):**
- Last-write-wins based on timestamp
- YouTube is treated as source of truth for inbound sync (if something was deleted on YouTube, we respect that even if our app still has it)
- Outbound sync failures don't block the UI; they retry in background

**API quota impact (per child, hourly sync):**
- Subscriptions.list: ~1-3 units (paginated if >50 subscriptions)
- PlaylistItems.list(LL): ~1-5 units (paginated)
- Playlists.list: ~1 unit
- PlaylistItems.list per playlist: ~1 unit each
- **Estimated: ~10-20 units/hour/child = 240-480 units/day/child**
- With 3 children: ~720-1,440 units/day (well within 10,000 limit)

### Usage Time Tracking
- Frontend sends heartbeat every 30 seconds while video is playing
- Backend records cumulative time per day
- When limit is reached, backend responds with "limit exceeded" on next heartbeat
- Frontend pauses video and shows message

### Token Refresh Strategy
- Store token expiry timestamp in DB
- Before any YouTube API call, check if token expires within 5 minutes
- If so, refresh proactively
- If refresh fails (revoked), mark account as needing re-auth

### Playlist YouTube Sync
When a child creates/modifies a playlist:
1. Action is performed locally in SQLite (immediate response to UI)
2. Background task calls YouTube Playlists API using child's OAuth tokens:
   - Create: `POST /youtube/v3/playlists` → store returned playlist ID
   - Add video: `POST /youtube/v3/playlistItems`
   - Remove video: `DELETE /youtube/v3/playlistItems`
   - Reorder: `PUT /youtube/v3/playlistItems` (update position)
3. Update `sync_status` column (synced/pending/error)
4. UI shows sync indicator; retries on failure with exponential backoff

### Cron System Design
- Uses `tokio-cron-scheduler` for in-process job scheduling
- Jobs are defined in the database (name, schedule, enabled flag)
- On server startup, scheduler loads all enabled jobs from DB
- Each job type maps to a Rust function (e.g., `update_ytdlp`)
- Job execution logs to `cron_job_runs` table (status, output, timing)
- API allows parents to trigger immediate execution, view history, toggle jobs, and change frequency
- Default jobs seeded on first run:
  - `ytdlp_update`: Check/update yt-dlp daily at 3:00 AM
  - `youtube_sync`: 2-way sync subscriptions, likes, playlists every hour
  - `cache_cleanup`: Remove cached segments for videos no longer on any child's allowlist (daily at 4:00 AM)

### Cron Schedule UI (Human-Readable Presets)
Instead of exposing raw crontab syntax, the UI presents a dropdown of preset frequencies per job. Each preset maps to a cron expression internally.

**Available presets:**
| Label | Cron Expression | Suitable For |
|-------|----------------|--------------|
| Every 15 minutes | `*/15 * * * *` | YouTube sync (aggressive) |
| Every 30 minutes | `*/30 * * * *` | YouTube sync |
| Every hour | `0 * * * *` | YouTube sync (default) |

**UI behavior:**

### Offline Video Download Flow (Client-Side)
```
Child taps "Download" on a video →
  Client shows quality picker (360p, 480p, 720p, 1080p) →
  Client calls POST /api/downloads (video_id, quality) →
  Server validates allowlist, returns download stream URL →
  Client fetches /api/downloads/:videoId/stream?quality=720p →
    If Background Fetch supported (Chromium):
      Use backgroundFetch.fetch() — download continues even if tab closes
      OS shows download progress notification
    Else (Firefox/Safari):
      Use regular fetch() with ReadableStream
      Show in-app progress bar (requires page to stay open)
  →
  Response body piped directly into OPFS via FileSystemWritableFileStream →
  On complete: client calls PUT /api/downloads/:videoId {status: "complete"} →
  Video now available offline via Service Worker
```

**Service Worker offline playback:**
- Intercepts requests to `/api/proxy/video` and `/api/downloads/:videoId/stream`
- Checks OPFS for cached video matching the video ID
- If found: serves from OPFS with full range request support (seeking)
- If not found and offline: shows "not available offline" message
- If not found and online: passes through to network

**Storage management:**
- `navigator.storage.persist()` called on first download to prevent eviction
- `navigator.storage.estimate()` checked before each download; warn if <500MB remaining
- Downloads UI shows total storage used and available
- If device storage is critically low, block new downloads and show a clear message

**Parent controls:**
- Parent can enable/disable offline downloading per child (stored in usage_limits or a per-child setting)
- When disabled, the "Download" button is hidden and download API returns 403 for that child

### Server-Side Two-Layer Caching

**Layer 1: Metadata/URL Cache (in-memory, ~4 hour TTL)**
```
Client requests /api/videos/:id/stream →
  Check in-memory cache for yt-dlp extraction results →
  If cached and not expired:
    Return cached format list + DASH manifest URL instantly (no subprocess call)
  If not cached:
    Call yt-dlp --dump-json to extract formats/manifest →
    Store result in memory cache with 4-hour TTL →
    Return formats to client
```
This avoids re-running the yt-dlp subprocess for repeat plays within the TTL window.

**Layer 2: Segment Cache (on-disk, LRU eviction)**
Videos are served using DASH adaptive streaming. YouTube splits videos into small segments (2-10 seconds, ~2-5 MB each). We cache at the segment level:

```
Client's vidstack player requests a DASH segment via our proxy →
  /api/proxy/segment?video_id=X&format=Y&segment=Z →
  Check segment cache directory for file →
  If cached:
    Serve from disk instantly →
    Update last_accessed_at
  If not cached:
    Fetch segment from YouTube (googlevideo.com) →
    Tee: stream to client AND write to disk simultaneously →
    Register segment in cache index
  If cache exceeds max size:
    Evict least-recently-accessed segments until under limit
```

**DASH manifest rewriting flow:**
```
Server gets DASH manifest from YouTube (via yt-dlp extraction) →
  Parse XML manifest →
  Rewrite all segment URLs to point through our proxy:
    Original: https://rr1---sn-xxx.googlevideo.com/videoplayback?...&sq=42
    Rewritten: /api/proxy/segment?video_id=X&format=137&sq=42&sig=<hmac>
  →
  Return rewritten manifest to client →
  vidstack player fetches segments through our proxy as needed
```

**Why segment-level caching is ideal for a kids' app:**
- Kids re-watch the same videos constantly — segments serve instantly on repeat
- Only watched portions get cached (partial views don't waste disk space)
- Siblings watching the same video get instant cache hits after first viewer
- Small eviction units (~2-5 MB segments) allow fine-grained LRU
- DASH provides natural quality adaptation — can cache segments at multiple qualities
- Audio segments cached separately (efficient for audio-only mode)

**Cache configuration (stored in app_config, managed via parent UI):**
- `cache_max_size`: Maximum total segment cache size (human-readable presets, see below)
- `metadata_cache_ttl_hours`: How long yt-dlp results stay in memory (default: 4)
- Cache directory: `/app/data/segment_cache/` (inside the Docker volume)
- Directory structure: `segment_cache/{video_id}/{format_id}/{segment_number}`

**Cache size presets (human-readable dropdown in parent UI):**
| Label | Value | Description |
|-------|-------|-------------|
| 5 GB | `5` | Minimal — caches ~10-20 videos worth of segments |
| 10 GB | `10` | Small — good for a few frequently-watched videos |
| 25 GB | `25` | Medium — covers most regular viewing |
| 50 GB (default) | `50` | Large — extensive caching for a family |
| 100 GB | `100` | Very large — rarely needs to evict |
| Unlimited | `0` | No size limit — cache grows until disk is full (uses `navigator.storage.estimate()` server-side disk check to warn when <10% free) |

When the cache exceeds the configured max size (unless unlimited), LRU eviction removes the least-recently-accessed segments until the cache is back under the limit. The parent UI shows current cache usage vs limit with a visual progress bar.

**Cache benefits:**
- Eliminates repeated yt-dlp extraction for popular videos (metadata cache)
- Eliminates re-downloading from YouTube for watched segments (segment cache)
- Near-instant playback for repeat views (kids' most common pattern)
- Offline downloads can be assembled from cached segments (no re-fetch needed)
- Only the watched portions consume disk space

### Cache Cleanup Flow
```
For each unique video_id in segment_cache table:
  Check: is this video_id in allowlisted_videos for ANY child?
  Check: is this video_id from a channel in allowlisted_channels for ANY child?
  Check: is this video_id in a playlist in allowlisted_playlists for ANY child?
  If none of the above:
    Delete all segment files for this video_id from disk →
    Remove rows from segment_cache table →
    Remove row from video_metadata_cache table →
    Log: "Evicted {video_id}: no longer allowlisted"
```

**Notes:**
- Runs daily by default (4:00 AM, after yt-dlp update job)
- Only removes segments for videos that are NOT on any child's allowlist
- A video that was previously allowlisted (and cached) but later removed from all allowlists will be cleaned up
- Does NOT remove segments for videos in blocked_videos (those are still allowlisted, just blocked from viewing)
- Parent can trigger manual cleanup via "Run Now" in the cron UI
- Cleanup respects the cache size setting — if cache is under the limit, cleanup still removes un-allowlisted content to keep the cache relevant

### yt-dlp Update Flow
```
Check GitHub releases API for yt-dlp/yt-dlp →
  Compare latest tag with current_version in DB →
  If newer:
    Download new binary to temp path →
    Verify with `--version` call →
    Atomically replace old binary (rename) →
    Update DB with new version →
    Log success
  If same: Log "already up to date"
  On failure: Log error, keep old binary
```

## Development Environment (Tilt)

The dev environment uses [Tilt](https://tilt.dev/) to orchestrate all services with live reload.

**Tiltfile** manages:
- Rust backend: `cargo-watch` rebuilds and restarts the server on any `src/`, `templates/`, or `Cargo.toml` change
- Vite frontend: runs in watch mode, rebuilds component bundles on file save
- SQLite database: auto-runs migrations when `migrations/` changes
- yt-dlp: verifies binary is available, warns if missing

```python
# Tiltfile (simplified)

# Backend: auto-rebuild + restart on Rust source or template changes
local_resource(
    'backend',
    serve_cmd='cargo watch -w src -w templates -x run',
    deps=['src/', 'templates/', 'Cargo.toml', 'migrations/'],
    labels=['server'],
)

# Frontend: Vite watch mode (rebuilds component bundles on change)
local_resource(
    'frontend',
    serve_cmd='npm run dev',
    resource_deps=['backend'],
    dir='frontend',
    labels=['client'],
)

# Run migrations when schema files change
local_resource(
    'migrations',
    cmd='sqlx migrate run',
    deps=['migrations/'],
    labels=['database'],
    resource_deps=['backend'],
)

# Verify yt-dlp is available
local_resource(
    'ytdlp-check',
    cmd='yt-dlp --version || echo "WARNING: yt-dlp not found"',
    labels=['deps'],
)
```

**Dev workflow:**
1. `nvm use` (reads `.nvmrc`, ensures correct Node version)
2. `cd frontend && npm install` (first time)
3. `tilt up` (starts everything)
4. App available at `http://localhost:3000` (Axum serves HTML pages + static assets)
5. Edit a `.ts` component → Vite rebuilds bundle → browser refresh picks it up
6. Edit a `.rs` file or template → cargo-watch rebuilds and restarts server (~2-5s)
7. Edit a `migrations/*.sql` file → sqlx auto-runs migration

5. **Publish** — Build & push Docker image to GitHub Container Registry (main branch only)

## Limitations & Future Considerations

- **yt-dlp dependency**: Stream extraction may break temporarily when YouTube updates. The daily auto-update cron job mitigates this, but there may still be brief windows of breakage.
- **API quotas**: YouTube Data API has a 10,000 unit/day quota. Caching is critical.
- **Video proxy bandwidth**: All video bytes flow through the server (YouTube → server → client). On a LAN this is a non-issue, but if clients access the server remotely (e.g., away from home), the server's outbound/egress bandwidth must support the video bitrate (~5-8 Mbps per 1080p stream).
- **Offline download limits**: OPFS storage varies by browser/device. Safari in private browsing disables OPFS entirely. Background Fetch (download while tab closed) is Chromium-only.
- **Single family**: One instance = one family. No multi-tenancy.
- **YouTube playlist sync**: YouTube API has rate limits on playlist operations. Batch operations may need throttling.
